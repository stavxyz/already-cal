# Dropbox Image Support & Generalized Attachment Extraction

**Date**: 2026-03-24
**Status**: Approved

## Problem

og-cal extracts images from Google Drive URLs in event descriptions but doesn't handle Dropbox share links. A live event on nobigbendwall.org has three Dropbox image URLs that render as broken images. Additionally, non-image file links (PDFs, docs, etc.) from any host are ignored entirely â€” the og-cal schema has an `attachments` field that's never populated from descriptions.

## Solution

Two changes:

1. **Extend image extraction** to recognize Dropbox share URLs and normalize them to direct-servable image URLs.
2. **Add generalized attachment extraction** that detects file URLs (PDF, doc, spreadsheet, zip, etc.) from any host and populates the `attachments` schema field. Render attachments in the detail view.

## Design

### Image extraction (images.js)

**New Dropbox patterns** alongside existing Drive support:
- `dropbox.com/scl/fi/` (current format)
- `dropbox.com/s/` (legacy format)
- `dl.dropboxusercontent.com` (already direct-serve, no normalization needed)

**Normalization rules**:
- For `dropbox.com` URLs: replace `dl=0` with `raw=1`. If the URL has an existing query string but no `dl=0`, append `&raw=1`. If no query string at all, append `?raw=1`. The `rlkey` parameter (required for `/scl/fi/` URLs) must be preserved.
- For `dl.dropboxusercontent.com` URLs: pass through as-is (already direct-serve).

**Image vs. non-image filtering**:
- Extension is extracted from the **URL path** (last path segment before query string), not from the full URL, to avoid false matches on query parameter values.
- Dropbox URLs with image extensions (`.jpg`, `.jpeg`, `.png`, `.gif`, `.webp`) â†’ extracted as images
- Dropbox URLs with known non-image extensions (`.pdf`, `.docx`, `.zip`, etc.) â†’ skipped, picked up by attachment extraction
- Dropbox URLs with no recognizable extension â†’ optimistic image treatment (try rendering, per-image `onerror` handler hides individual failures)

`normalizeImageUrl()` is extended to handle Drive, Dropbox, and `dl.dropboxusercontent.com` URLs.

**Per-image error handling**: Currently only the first image in the gallery has an `onerror` handler that removes the gallery. This should be extended so each `<img>` gets its own `onerror` that hides just that image (or removes it from the gallery), so a non-image URL in position 2+ doesn't show a broken icon.

### Attachment extraction (new: attachments.js)

A new `extractAttachments()` function, **host-agnostic**:

**Detection**: Scans description for URLs whose **path** (last segment before query string) contains a known file extension:
- Documents: `.pdf`, `.doc`, `.docx`
- Spreadsheets: `.xls`, `.xlsx`, `.csv`
- Presentations: `.ppt`, `.pptx`
- Archives: `.zip`
- Text: `.txt`

Works regardless of host â€” Dropbox, Drive, S3, any raw URL. Dropbox is intentionally **not** added as a platform in `links.js`; file URLs from Dropbox (and all other hosts) flow naturally to attachment extraction via the cascading pipeline.

**Labeling**: Labels generated by broad file type category:
- `"Download PDF"`, `"Download Document"`, `"Download Spreadsheet"`, `"Download Presentation"`, `"Download Archive"`, `"Download File"`

**URL normalization for known hosts**:
- Dropbox: `dl=0` â†’ `raw=1` for direct download (same rules as image normalization, including `rlkey` preservation). `dl.dropboxusercontent.com` passed through as-is.
- Google Drive: `/uc?export=download&id={fileId}` variant for non-image files
- All other hosts: URL passed through as-is

**Schema**: Populates the existing `attachments` field, normalized to a consistent shape:
```json
{
  "label": "Download PDF",
  "url": "https://www.dropbox.com/scl/fi/.../flyer.pdf?rlkey=...&raw=1",
  "type": "pdf"
}
```

**Reconciling with Google Calendar API attachments**: `transformGoogleEvents()` already populates `attachments` from `item.attachments` using `{title, url, mimeType}`. These should be normalized to the same `{label, url, type}` schema. Map `title` â†’ `label` (or generate from `mimeType` if title is generic), `fileUrl` â†’ `url`, and derive `type` from `mimeType`. Description-extracted attachments are then appended after the API-provided ones.

**Stripping**: Extracted attachment URLs removed from description text, same `stripUrl()` pattern as images and links.

### Data flow (data.js)

**Execution order** in event enrichment:
1. `extractImage()` â€” images (including Dropbox/Drive image URLs)
2. `extractLinks()` â€” platform links (Eventbrite, Instagram, etc.)
3. `extractAttachments()` â€” remaining file URLs as attachments

Each function receives the already-cleaned description from the previous step, preventing double-extraction.

**Dual enrichment paths**: Both `enrichEvent()` and `transformGoogleEvents()` currently run image/link extraction independently. Refactor `transformGoogleEvents()` to delegate enrichment to `enrichEvent()` â€” the transform builds the base event shape, then `enrichEvent()` handles all description extraction (images, links, attachments) consistently in one place.

### Detail view UI (views/detail.js)

- New section below description, above existing platform link buttons
- CSS class: `ogcal-detail-attachments` (consistent with `ogcal-detail-links`, `ogcal-detail-description`, etc.)
- Each attachment rendered as a button/link, same visual style as platform links
- Text label indicating file type (e.g., "Download PDF") â€” no icon library
- Opens in new tab (`target="_blank"`)
- Section omitted when no attachments (same pattern as links)

### Testing

- `test/images.test.js` â€” Dropbox image URL extraction and normalization (all three URL formats, `rlkey` preservation, extension filtering, optimistic extensionless handling, dedup, `dl.dropboxusercontent.com` passthrough)
- `test/attachments.test.js` â€” new file: PDF extraction, docx, zip, mixed hosts, Dropbox normalization, Drive normalization, deduplication with already-extracted images/links, labeling, Google Calendar API attachment normalization, edge cases

## Out of Scope

- PDF thumbnail previews or inline rendering
- File type icons (icon library)
- Download progress indicators
- OneDrive/Box/S3-specific URL normalization (URLs pass through as-is, which works fine)
